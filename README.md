# impro-dev

# Для разработчиков о проекте

Для тестирования и проверки на синтаксические ошибки использован ИИ. При работе с проектом, используя ИИ был создан файл с требованиями, которые должен использовать ИИ. Для ознакомления открыть файл `AGENTS.md`.

## Локальная настройка Codex MCP

Файл `.codex/config.toml` игнорируется Git и хранит только локальные настройки.

Перед работой с MCP-серверами (например, Playwright) сделайте один раз:

1. Скопируйте `.codex/config.example.toml` в `.codex/config.toml`.
2. При необходимости дополните локальный `.codex/config.toml` своими MCP-серверами.

Важно:

- изменения в `.codex/config.toml` не попадут в репозиторий;
- общий шаблон и договоренности команды хранятся в `.codex/config.example.toml`.

## Команды проверки и форматирования

Проект использует Prettier, ESLint, Stylelint и HTMLHint.

### `npm run format`

Безопасный форматтер для исходников проекта. Форматирует только:

- `README.md`
- корневые конфиги `*.json`, `*.js`, `*.mjs`
- страницы `src/*.html`
- SCSS-исходники `src/assets/css/**/*.scss`
- основной JS `src/assets/js/app.js`

Что важно:

- команда не трогает vendor/минифицированные файлы и сборочные артефакты;
- исключения лежат в `.prettierignore` (`js-includes`, `css-includes`, `dist`, `*.css`, `*.css.map`, `package-lock.json`).

Влияние на код:

- Prettier меняет форматирование (отступы, переносы, кавычки, запятые и т.д.);
- бизнес-логику и поведение приложения менять не должен.

### `npm run lint:js`

Проверяет JavaScript в `src/assets/js/**/*.js` через ESLint.

Сейчас включены только критичные для работы правила:

- `no-undef` (использование не объявленных переменных)
- `no-unreachable` (недостижимый код)
- `no-dupe-keys` (дубли ключей в объектах)
- `no-redeclare` (повторное объявление переменных)

Важно:

- браузерные и библиотечные глобалы (`window`, `document`, `$`, `WOW`, `Typed` и т.д.) заранее объявлены в ESLint-конфиге, чтобы не было лишнего шума.

### `npm run lint:scss`

Проверяет SCSS в `src/assets/css/**/*.scss` через Stylelint.

Сейчас режим проверки максимально практичный:

- включена только критичная проверка `color-no-invalid-hex` (битые hex-цвета)

Важно:

- Stylelint использует `postcss-scss`, чтобы корректно парсить SCSS и не выдавать ложные CSS-синтаксические ошибки.

### `npm run lint:html`

Проверяет только итоговые страницы `src/*.html` через HTMLHint (без шаблонных partial/layout файлов).

Что ловит:

- ошибки парности тегов
- проблемы с регистром тегов/атрибутов
- некорректные кавычки атрибутов

Важно:

- правило `id-unique` отключено, чтобы не шуметь на повторяющиеся `id` в текущей архитектуре шаблонов.

### `npm run lint:html:dist`

Проверяет собранные страницы в `dist/**/*.html`.

Это полезно как финальная проверка результата после сборки.

### `npm run check`

Запускает все проверки подряд:

1. `lint:scss`
2. `lint:js`
3. `lint:html`

Если хотя бы одна проверка не проходит, команда завершается с ошибкой.
