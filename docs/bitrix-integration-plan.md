# План интеграции с Bitrix

## Цели
- Сохранить текущее визуальное поведение и адаптивную верстку.
- Разделить страницу на переиспользуемые фрагменты шаблонов.
- Перенести обработку формы на backend-эндпоинт и оставить клиентскую валидацию как progressive enhancement.

## Карта блоков страницы Promotion
- `data-bx-block="promo-hero"`: hero-область с хлебными крошками, тегами, заголовком, CTA и логотипами.
- `data-bx-block="promo-services"`: сетка карточек услуг и стоимости (`dev-vars`).
- `data-bx-block="promo-booking-hero"`: секция CTA/бронирования в середине страницы.
- `data-bx-block="promo-features"`: список преимуществ (`dev-features`).

## Рекомендуемая структура Bitrix
- `header.php` / `footer.php`: глобальный макет.
- `local/templates/<template_name>/components/`: переиспользуемые шаблоны блоков.
- `local/templates/<template_name>/assets/`: текущие собранные CSS/JS/изображения.
- `local/templates/<template_name>/include/`: статические текстовые фрагменты, редактируемые из CMS.

## Примечания по миграции
- Сохраняйте существующие BEM-классы без изменений, чтобы избежать регрессий.
- Используйте маркеры `data-bx-block` для детерминированного серверного подключения блоков.
- Замените inline-заглушку отправки формы на компонент формы Bitrix или endpoint контроллера.
- Сохраняйте JS-проверки (`if element exists`) для корректного рендера частичных шаблонов.
